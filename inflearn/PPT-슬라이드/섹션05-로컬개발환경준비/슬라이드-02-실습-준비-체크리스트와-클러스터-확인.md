# 슬라이드 02: 실습 준비 체크리스트와 클러스터 확인

## 슬라이드 내용 (한 장)

**공통 체크**  
• K8s 클러스터 접근 가능 (kubectl get nodes 성공)  
• KUBECONFIG가 올바른 클러스터를 가리킴  
• Git으로 tz-chatbot 클론 완료  
• (선택) Helm 설치 — bootstrap·RAG·Dify 설치 시 사용

**로컬 gateway/admin 실행 시**  
• Python 3.11+ / .env 또는 환경변수 (DIFY_*, CHAT_GATEWAY_JWT_SECRET 등) / (선택) 가상환경·pip install -r requirements.txt

**K8s 배포 실습 시**  
• 네임스페이스 생성 가능 또는 devops·rag·dify 존재  
• RAG Secret 준비 (MinIO 접근 키, Gemini 또는 OpenAI API Key)  
• DB: chat_admin용 PostgreSQL·DB 생성·접속 정보 (README Database setup)

**클러스터 접근 확인**: kubectl config current-context, kubectl get nodes, kubectl get ns. bootstrap Phase 0에서도 get nodes 실행 후 실패 시 중단.

---

## 발표 노트

실습을 시작하기 전에 체크리스트를 한 번 확인하시면 좋습니다. 공통으로는 Kubernetes 클러스터에 접근이 되는지, 즉 kubectl get nodes가 성공하는지, KUBECONFIG가 사용하실 클러스터를 가리키는지, Git으로 tz-chatbot 저장소 클론이 끝났는지 봅니다. Helm은 선택이지만, bootstrap이나 RAG, Dify 설치할 때 쓰이니까 있으면 좋습니다.

로컬에서 gateway나 admin을 실행하실 때는 Python 3.11 이상이 필요하고, DIFY_BASE_URL, DIFY_API_KEY, CHAT_GATEWAY_JWT_SECRET 같은 값을 .env나 환경변수로 설정하셔야 합니다. 가상환경을 만들고 pip install -r requirements.txt 해 두시면 됩니다.

K8s에 실제로 배포하는 실습을 하실 때는, 네임스페이스를 만들 수 있거나 이미 devops, rag, dify가 있어야 하고, RAG용 Secret을 준비해야 합니다. MinIO 접근 키와 Gemini 또는 OpenAI API Key가 필요하고, rag/README에 나온 대로 만드시면 됩니다. chat_admin DB는 PostgreSQL을 쓰실 때 미리 생성하고 접속 정보를 준비하시는데, README의 Database setup을 보시면 됩니다.

클러스터 접근 확인은 kubectl config current-context로 현재 컨텍스트를 보고, kubectl get nodes, kubectl get ns로 노드와 네임스페이스가 보이는지 확인하시면 됩니다. bootstrap.sh는 Phase 0에서 kubectl get nodes를 실행해서 실패하면 그대로 중단하니까, 미리 한 번 실행해 두시면 나중에 클러스터 접속 오류를 줄일 수 있습니다.
