# 슬라이드 02: 확장 주제와 추가 학습

## 슬라이드 내용 (한 장)

**확장 주제**  
• **새 토픽 추가**: MinIO 경로·Qdrant 컬렉션·Ingestion Job/CronJob·RAG Backend Deployment를 토픽별로 추가. docs/rag-multi-topic 패턴 참고  
• **하이브리드 검색**: 벡터 검색에 BM25 등 키워드 검색 혼합 (저장소·RAG Backend 구현 확인)  
• **증분 Ingestion**: INCREMENTAL=true로 변경분만 재색인. 대용량 문서 증가 시 유리  
• **모니터링·알림**: Prometheus·Grafana, Loki·로그 검색, 알림 룰  
• **다중 리전·클러스터**: 동일 구성을 여러 클러스터에 배포하거나 Dify·RAG만 별도 클러스터로 분리

**추가 학습 자료**  
• **Kubernetes**: 공식 문서, “바로 쓰는 CI/CD on K8s” 등 실습 강의  
• **RAG·LangChain**: LangChain 문서, RAG 설계 베스트 프랙티스  
• **Dify**: 공식 문서, 워크플로우·도구 연동  
• **PostgreSQL·MinIO·Qdrant**: 각 제품 문서(백업·튜닝·고가용성)

---

## 발표 노트

이 구조를 바탕으로 확장할 수 있는 주제를 몇 가지 정리했습니다. 새 토픽을 추가할 때는 MinIO 경로, Qdrant 컬렉션, Ingestion Job·CronJob, RAG Backend Deployment를 토픽마다 하나씩 추가하면 되고, docs의 rag-multi-topic 패턴을 그대로 따라 하시면 됩니다. 하이브리드 검색은 벡터 검색에 BM25 같은 키워드 검색을 섞는 건데, 저장소와 RAG Backend에서 어떻게 지원하는지 확인해서 적용하시면 됩니다. 증분 Ingestion은 INCREMENTAL=true로 두면 변경·추가분만 재색인해서 대용량 문서가 늘어날 때 유리합니다. 모니터링과 알림은 Prometheus·Grafana, Loki로 로그 검색, 알림 룰을 추가하는 방향입니다. 다중 리전이나 여러 클러스터를 쓰실 때는 같은 구성을 여러 클러스터에 배포하거나, Dify와 RAG만 별도 클러스터로 빼는 설계도 가능합니다.

추가 학습 자료는 Kubernetes는 공식 문서나 “바로 쓰는 CI/CD on K8s” 같은 실습 강의, RAG와 LangChain은 LangChain 문서와 RAG 설계 베스트 프랙티스, Dify는 공식 문서와 워크플로우·도구 연동, PostgreSQL·MinIO·Qdrant는 각 제품 문서에서 백업·튜닝·고가용성을 참고하시면 됩니다. 이 강의에서 다룬 tz-chatbot 구조를 기반으로 실제 서비스 규모와 요구사항에 맞게 확장해 보시기 바랍니다.
